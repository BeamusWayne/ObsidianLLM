## 数据流

### Clip & Ingest 流程

```
用户粘贴 URL
    │
    ▼
POST /api/clip
    │  trafilatura 抓取正文
    │  下载图片 → raw/images/
    │  保存 .md → raw/articles/
    │
    ▼
检查 settings.auto_ingest
    │
    ├── true  → worker.enqueue(path)   ← 立即返回给前端
    └── false → 只保存，等用户手动选择
                    │
                    ▼
              用户在 Raw Files 面板勾选文件
              点「处理选中」→ POST /api/ingest {files: [...]}
                    │
                    ▼
              worker.enqueue(paths)    ← 跳过 .ignored.json 中的文件
    │
    ▼
IngestWorker (后台线程)
    │
    ├─ route_backend(len(content))
    │      parallel.enabled = false → 直接用 fallback_chain[0]
    │      parallel.enabled = true  → 按字符数路由到 small/large_backend
    │
    ├─ llm.chat(messages, backend=X)
    │      按 fallback_chain 顺序尝试：
    │      [claude] → 成功 ✓
    │              → 余额不足 → warn_cb → [custom] → [ollama]
    │
    ├─ 解析 ![图片](ref) → describe_image() → 同样走 fallback_chain
    │
    ├─ 写入 wiki/*.md
    ├─ 更新 .processed.json
    └─ 持久化状态 → .ingest_status.json
```

### 状态轮询

```
前端每 3 秒 GET /api/ingest/status
    │
    ▼
返回 IngestWorker._status：
    running / current_file / queued
    log（最近 80 行）/ elapsed / billing_fallback
```

### 文件忽略流程

```
用户在 Raw Files 面板勾选文件 → 点「忽略」
    │
    ▼
POST /api/raw/ignore { paths: [...] }
    │  写入 .ignored.json
    │
    ▼
后续 Ingest（自动或手动）跳过这些文件

用户点「恢复」→ POST /api/raw/unignore
    │  从 .ignored.json 移除
    └─ 文件重新变为「待处理」
```

### IngestWorker（`scripts/ingest.py`）

常驻后台线程（`daemon=True`），服务器启动时自动创建。

```
状态机：
  idle ──► running ──► idle
              │
              └── error (last_error 记录，不停止 Worker)
```

关键机制：
- **防重复入队**：`_queued_set` 记录已在队列中的文件，同一文件不会入队两次
- **断点续传**：启动时调用 `enqueue_unprocessed()`，自动补入上次未完成的文件
- **忽略文件跳过**：入队时检查 `.ignored.json`，已忽略的文件不入队
- **锁保护**：`threading.Lock` 保护 `.processed.json` 和 `wiki/` 写入，防止并行竞态
- **状态持久化**：每次状态变更写入 `.ingest_status.json`，重启后恢复历史记录

### LLM 层（`scripts/llm.py`）

```
chat(messages, backend=None, warn_cb=None)
    │
    ├── 读取 settings.json（每次调用动态读，Settings 页改完立即生效）
    │
    ├── _build_chain(s, backend)
    │       backend 显式指定 → 放到链首，其余按 fallback_chain 补齐
    │       未指定 → 直接用 settings.fallback_chain
    │
    └── 按链顺序尝试每个 backend：
            _chat_single(s, "claude", messages)
                → 成功 → 返回
                → 余额不足 → warn_cb() → 继续下一个
                → 其他错误 → 直接 raise
            _chat_single(s, "custom", messages)  ← 如果在链中
            _chat_single(s, "ollama", messages)
            → 全部失败 → raise RuntimeError
```

同样的链式逻辑用于 `describe_image()`。

### 路由逻辑（`route_backend`，ingest.py）

```python
parallel.enabled = false  →  直接用 fallback_chain[0]（不路由）
parallel.enabled = true   →
    字符数 < size_threshold_chars  →  small_backend
    字符数 ≥ size_threshold_chars  →  large_backend
    （small/large_backend 本身也会走 fallback_chain）
```

## 并行处理

Worker 单线程顺序处理队列（一次一个文件）。大批量手动重跑时（`ingest --all`），`run_ingest()` 使用 `ThreadPoolExecutor`，workers 数量由 `settings.parallel.workers` 控制。

```
手动批量：ThreadPoolExecutor(workers=3)
             ├── 文件 A → fallback_chain[0]（如 Claude）
             ├── 文件 B → fallback_chain[0]
             └── 文件 C → fallback_chain[0]
             （并发，锁保护写入）

自动入队（Clip 后）：Worker 单线程队列
             文件 A → 文件 B → 文件 C（顺序）
```

## 关键文件说明

| 文件 | 自动维护 | 说明 |
|------|---------|------|
| `settings.json` | 否（用户配置） | LLM 后端、fallback_chain、auto_ingest、并行参数 |
| `schema.md` | 否（用户配置） | LLM 行为规范，改完需重跑 `--all` |
| `.processed.json` | 是 | 已处理文件路径列表，Worker 去重依据 |
| `.ignored.json` | 是（用户控制） | 用户标记忽略的文件，Ingest 时跳过 |
| `.ingest_status.json` | 是 | Worker 状态快照，重启后恢复 |
| `wiki/log.md` | 是 | 每次 ingest 追加记录（最新在前） |

## 技术选型说明

| 选择 | 原因 |
|------|------|
| FastAPI | 异步支持好，自动 OpenAPI 文档，Pydantic 数据校验 |
| Vue 3 + Vite | 轻量，无状态管理库（用 `ref` 够了），构建快 |
| D3.js 力导向图 | 原生 SVG，主题感知（CSS vars），无额外图库依赖 |
| `urllib`（非 requests）| 避免额外依赖，SSL 用 certifi 补全 |
| `threading.Thread`（非 asyncio Task）| ingest 是 CPU/IO 密集型阻塞操作，线程更自然 |
| `.processed.json`（非数据库）| 文件系统即状态，零依赖，Git 友好 |
| `fallback_chain`（链式 fallback）| 用户可自定义 backend 优先级和降级策略，解耦主模型与备用模型 |
